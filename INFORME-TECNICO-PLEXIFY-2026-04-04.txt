SECCION 1: ARQUITECTURA GENERAL
================================================================================

El sistema tiene 3 capas:

   [FRONTEND]              [BACKEND]               [BASE DE DATOS]
   React 19 + Vite    →    FastAPI (Python)    →    PostgreSQL
   GitHub Pages             Railway                  Railway
   SCSS BEM                 SQLAlchemy ORM           40 tablas
   164 archivos             104 archivos             pgvector

Flujo de datos:
1. Usuario abre el sitio → GitHub Pages sirve el SPA (React)
2. Si es ruta publica (/, /pricing, /lina-ia) → Landing Page
3. Si tiene token JWT → CRM Shell (app autenticada)
4. Frontend hace fetch a Railway API → FastAPI procesa → PostgreSQL
5. WhatsApp webhook → Meta envia mensajes → Backend procesa → Lina IA responde

SECCION 5: COMO FUNCIONA — FLUJOS PRINCIPALES
================================================================================


--- FLUJO 1: CLIENTE AGENDA POR WHATSAPP ---

1. Cliente envia "Quiero un corte para manana" por WhatsApp
2. Meta webhook → POST /api/whatsapp/webhook
3. Backend identifica tenant por wa_phone_number_id
4. Crea/actualiza WhatsAppConversation + WhatsAppMessage
5. Analiza sentimiento (rule-based, zero tokens)
6. Lanza ai_auto_reply() en background:
   a. Prompt injection check (37 patrones)
   b. Construye contexto: negocio, cliente, servicios, agenda 3 dias
   c. Inyecta estado ABIERTO/CERRADO (calculado por codigo)
   d. Inyecta memorias del cliente (pgvector)
   e. Llama Claude Sonnet con 48 acciones disponibles
   f. Claude responde + ejecuta create_appointment con conflicto check
   g. Si conflicto → segunda llamada IA con alternativas
   h. Envia respuesta por WhatsApp (Meta API)
   i. Guarda mensaje outbound en DB
   j. Extrae memorias de la conversacion (background)
   k. Detecta tareas pendientes (follow-up, confirmacion)
7. Cliente recibe respuesta en 3-5 segundos


--- FLUJO 2: ADMIN COBRA UN SERVICIO (CHECKOUT) ---

1. Admin va a Agenda → click en cita → "Cobrar"
2. CheckoutModal se abre (wizard 4 pasos)
3. Paso 1: Servicios (auto-llenado desde la cita, puede agregar mas)
4. Paso 2: Descuento (% o fijo) + propina
5. Paso 3: Metodo de pago (efectivo/Nequi/Daviplata/tarjeta/mixto)
6. Paso 4: Confirmar → POST /api/checkout
7. Backend crea:
   - Checkout record
   - VisitHistory (1 por staff)
   - Invoice + InvoiceItems
   - Appointment.status = "paid"
   - Puntos de lealtad al cliente
8. Frontend muestra confirmacion


--- FLUJO 3: NUEVO NEGOCIO SE REGISTRA ---

1. Dueno va al landing → "Empiece gratis"
2. Wizard de 7 pasos:
   - Tipo de negocio (20 opciones)
   - Datos del negocio (nombre, direccion, telefono)
   - Plan (Starter/Pro/Business)
   - Servicios (templates por tipo de negocio)
   - Staff (agregar miembros)
   - Datos del admin (usuario, password)
   - Resumen y confirmar
3. POST /api/public/register
4. Backend crea: Tenant + Admin + Services + Staff + Location + AIConfig
5. Admin puede hacer login inmediatamente


--- FLUJO 4: SCHEDULER (cada 2 minutos) ---

1. scheduler.py loop infinito
2. Cada 2 min:
   - Ejecuta automation_engine: evalua 29 triggers para cada tenant
   - Recordatorios de citas (24h, 1h antes)
   - Birthday check → bonus de lealtad
   - Sweep de tareas pendientes
   - Morning review (9am)
   - No-show follow-up
   - Token health check
   - Daily analytics

SECCION 7: SEGURIDAD
================================================================================

- JWT con cookies httponly (no localStorage)
- Single-device sessions (refresh token actualiza DB)
- Tenant isolation en TODAS las queries (safe_tid)
- Prompt injection protection (37 patrones)
- Core safety prompt inmutable (11 reglas)
- Client privacy: agenda muestra [Ocupado] a otros clientes
- Rate limiter: 3 msgs/hora, 6/dia por conversacion
- Private keys mascaradas en GET responses
- CORS configurado por dominio
- Password policy en registro
