# AlPelo CRM — Plan Completo: De Prototipo a Producción

## Contexto
El frontend de AlPelo CRM está construido (React 19 + Vite + SCSS). Tiene 3 módulos sólidos (Login, Dashboard, CRM Clientes) y 3 funcionales como demo (Citas, Mensajería, Chat IA). Todo corre con datos mock y localStorage. No hay backend, ni base de datos, ni APIs reales.

**Objetivo:** Llevar el proyecto a producción con backend real, base de datos, IA, integración WhatsApp/Meta, y deploy en VPS con Docker.

**Stack elegido:**
- Backend: Python + FastAPI
- Base de datos: PostgreSQL
- IA: Claude API (Anthropic)
- Mensajería: WhatsApp Business API (Meta)
- Deploy: VPS + Docker Compose

### 1.2 — Diseño de Base de Datos PostgreSQL

**Tablas principales:**

| Tabla | Campos clave | Relaciones |
|-------|-------------|------------|
| `users` | id, name, email, password_hash, role (admin/barber), phone, avatar_url, is_active | → appointments, → visits |
| `clients` | id, name, phone, email, birthday, gender, source, accepts_whatsapp, tags[], notes, loyalty_points, created_at | → visits, → appointments |
| `staff` | id, user_id (nullable), name, role (barbero/estilista/manicurista), specialty, phone, is_active, rating | → appointments, → visits |
| `services` | id, name, category, price, duration_minutes, is_active | → visits, → appointments |
| `appointments` | id, client_id, staff_id, service_id, date, start_time, end_time, status (pending/confirmed/cancelled/completed), notes | FK → clients, staff, services |
| `visits` | id, client_id, staff_id, service_id, date, amount, tip, rating, notes | FK → clients, staff, services |
| `campaigns` | id, name, template, segment, status (draft/sent/scheduled), sent_count, created_by, created_at | → messages |
| `messages` | id, campaign_id, client_id, content, channel (whatsapp/sms), status (sent/delivered/read/failed), sent_at | FK → campaigns, clients |
| `chat_history` | id, user_id, role (user/assistant), content, created_at | FK → users |
| `notifications` | id, user_id, type, title, message, is_read, created_at | FK → users |

**Índices importantes:**
- `clients(phone)` — UNIQUE, búsqueda rápida
- `appointments(date, staff_id)` — Agenda por barbero por día
- `visits(client_id, date)` — Historial por cliente
- `messages(campaign_id, status)` — Tracking de campañas

### 1.5 — Pasos concretos

1. **Instalar PostgreSQL** localmente o usar Docker (`docker run -d -p 5432:5432 -e POSTGRES_PASSWORD=secret postgres:16`)
2. **Crear el proyecto FastAPI** con la estructura de arriba
3. **Definir los modelos SQLAlchemy** — empezar por `users`, `clients`, `services`, `staff`
4. **Configurar Alembic** para migraciones (`alembic init alembic`, configurar `database.py`)
5. **Crear la primera migración** (`alembic revision --autogenerate -m "initial tables"`)
6. **Implementar auth** (login, JWT, middleware) — esto es lo primero porque todo lo demás lo necesita
7. **Implementar endpoints de clientes** — conectar con la página de CRM que ya funciona
8. **Migrar el motor de estados** (`clientStatus.js` → `status_engine.py`) — misma lógica, en Python
9. **Implementar endpoints de citas** — conectar con la página de Appointments
10. **Implementar endpoints de staff y dashboard**
11. **Semillar la BD** con los datos que ya están en `mockData.js` (30 clientes, 18 staff, 20 servicios, 127 visitas)

**Tiempo estimado: 2-3 semanas** (trabajando a buen ritmo)

## FASE 3: Integración WhatsApp Business API (Semana 4-5)

### 3.1 — Qué necesitas

| Requisito | Detalle |
|-----------|---------|
| **Cuenta Meta Business** | Crear en business.facebook.com |
| **App en Meta Developers** | Crear en developers.facebook.com |
| **WhatsApp Business API** | Activar en la app de Meta |
| **Número de teléfono** | Un número dedicado para el negocio (no puede ser el personal) |
| **Verificación del negocio** | Meta verifica que AlPelo es un negocio real |

### 3.2 — Proceso paso a paso

1. **Crear cuenta Meta Business Manager** en business.facebook.com
   - Necesitas: nombre del negocio, dirección, documento de identidad del dueño
   - Tiempo de verificación: 2-7 días

2. **Crear app en Meta for Developers** (developers.facebook.com)
   - Tipo: Business
   - Agregar producto: WhatsApp
   - Te dan un **Phone Number ID** y un **Access Token** temporal

3. **Registrar número de teléfono**
   - Puedes usar el número de prueba gratuito de Meta (para desarrollo)
   - Para producción: registrar el número real de AlPelo
   - Verificación por SMS o llamada

4. **Crear plantillas de mensaje**
   - Meta EXIGE que los mensajes de inicio de conversación usen plantillas aprobadas
   - Las plantillas se crean en Meta Business Manager
   - Tiempo de aprobación: 24-48 horas
   - Ejemplo de plantilla para confirmación de cita:
     ```
     Hola {{1}}, tu cita en AlPelo para {{2}} el {{3}} a las {{4}}
     ha sido actualizada. ¿Confirmas? Responde SI o NO.
     ```

5. **Configurar Webhook** en el backend
   - Meta envía notificaciones cuando el cliente responde
   - Endpoint: `POST /api/webhooks/whatsapp`
   - Verificación: `GET /api/webhooks/whatsapp` (challenge de Meta)
   - Esto permite: saber cuando el cliente lee el mensaje, cuando responde "SI" para confirmar

### 3.3 — Flujo de mensajería

```
[Admin crea campaña en frontend]
       ↓
[Frontend envía a POST /api/messaging/bulk]
       ↓
[Backend recibe lista de clientes + plantilla]
       ↓
[Backend envía a WhatsApp Cloud API por cada cliente]
  POST https://graph.facebook.com/v21.0/{phone_number_id}/messages
       ↓
[Meta entrega el mensaje al WhatsApp del cliente]
       ↓
[Cliente responde → Meta envía webhook al backend]
       ↓
[Backend actualiza estado del mensaje/cita]
       ↓
[Frontend muestra estado actualizado]
```

### 3.4 — Costos de WhatsApp Business API

| Concepto | Costo |
|----------|-------|
| Conversaciones de servicio (cliente inicia) | GRATIS (primeras 1,000/mes) |
| Conversaciones de marketing (negocio inicia) | ~$0.015 USD por mensaje (Colombia) |
| 500 mensajes masivos/mes | ~$7.50 USD/mes |
| 2,000 mensajes masivos/mes | ~$30 USD/mes |

### 3.5 — Librería Python recomendada

Usar `httpx` directamente con la API REST de Meta. No necesitas librería especial, la API es sencilla:
- Enviar mensaje: `POST` a `graph.facebook.com`
- Recibir respuesta: webhook `POST` a tu servidor

**Tiempo estimado: 1-2 semanas** (incluye aprobación de Meta)

## FASE 4: Integración IA con Claude API (Semana 5-6)

### 4.1 — Qué necesitas

| Requisito | Detalle |
|-----------|---------|
| **Cuenta Anthropic** | Crear en console.anthropic.com |
| **API Key** | Se genera en la consola |
| **Créditos** | Prepago, empezar con $5-10 USD |

### 4.2 — Cómo funciona la integración

El chat de IA NO habla directamente con Claude. El backend actúa de intermediario y le da CONTEXTO sobre el negocio:

```
[Usuario pregunta: "¿Cuáles clientes están en riesgo?"]
       ↓
[Frontend envía a POST /api/chat/message]
       ↓
[Backend recibe la pregunta]
       ↓
[Backend consulta la BD: clientes en_riesgo, sus datos, historial]
       ↓
[Backend construye el PROMPT para Claude:]
  "Eres el asistente de AlPelo Peluquería en Bucaramanga.
   Datos del negocio: 30 clientes, 18 staff, 20 servicios...

   Clientes en riesgo actualmente:
   - Juan Pérez: 52 días sin visita, 8 visitas totales, $450,000 gastados
   - María López: 48 días sin visita, 3 visitas totales, $120,000 gastados

   El usuario pregunta: ¿Cuáles clientes están en riesgo?

   Responde en español colombiano, sé específico con los datos."
       ↓
[Claude API responde con análisis contextualizado]
       ↓
[Backend guarda en chat_history y retorna al frontend]
       ↓
[Frontend muestra la respuesta con formato]
```

### 4.3 — Modelo recomendado

| Modelo | Costo (input/output por millón tokens) | Uso |
|--------|---------------------------------------|-----|
| **Claude Haiku 4.5** | $0.80 / $4.00 | Consultas rápidas, día a día — RECOMENDADO |
| Claude Sonnet 4.6 | $3.00 / $15.00 | Análisis complejos (opcional) |

Con Haiku: una consulta típica cuesta ~$0.001-0.003 USD. Con 100 consultas/día = ~$0.30 USD/día = ~$9 USD/mes.

### 4.4 — Tipos de consultas que el asistente debe manejar

1. **Consultas sobre clientes**: "¿Quiénes no vienen hace más de 30 días?"
2. **Análisis de negocio**: "¿Cuál es mi servicio más rentable?"
3. **Recomendaciones**: "¿Qué campaña debería enviar esta semana?"
4. **Datos específicos**: "¿Cuánto ha gastado Carlos Mendoza este año?"
5. **Tendencias**: "¿Cómo van los ingresos comparado con el mes pasado?"

El backend consulta la BD según el tipo de pregunta y le pasa los datos relevantes a Claude. Claude analiza y responde.

### 4.5 — Librería Python

```
pip install anthropic
```

**Tiempo estimado: 1 semana**

## FASE 5: Docker Compose (Semana 6-7)

### 5.1 — Arquitectura Docker

```
docker-compose.yml
├── frontend      (Nginx sirviendo el build de React)
├── backend       (FastAPI + Uvicorn)
├── postgres      (PostgreSQL 16)
├── redis         (Cache + colas de mensajes) [opcional pero recomendado]
└── nginx         (Reverse proxy + SSL)
```

### 5.2 — Servicios y puertos

| Servicio | Puerto interno | Puerto externo | Imagen |
|----------|---------------|----------------|--------|
| frontend | 80 | — | nginx:alpine + build de React |
| backend | 8000 | — | python:3.11-slim + FastAPI |
| postgres | 5432 | — | postgres:16-alpine |
| redis | 6379 | — | redis:7-alpine |
| nginx (proxy) | 80, 443 | 80, 443 | nginx:alpine |

Solo nginx expone puertos al exterior. Todo lo demás es interno.

### 5.3 — Archivos Docker necesarios

**`frontend/Dockerfile`** — Multi-stage build:
- Stage 1: `node:20-alpine` → `npm ci && npm run build`
- Stage 2: `nginx:alpine` → copia el `dist/` y un `nginx.conf`

**`backend/Dockerfile`**:
- Base: `python:3.11-slim`
- Copia `requirements.txt`, instala dependencias
- Copia código, expone 8000
- CMD: `uvicorn app.main:app --host 0.0.0.0 --port 8000`

**`nginx/nginx.conf`** — Reverse proxy:
- `/` → frontend (archivos estáticos)
- `/api/` → backend:8000
- SSL con Let's Encrypt (certbot)

**`docker-compose.yml`** — Orquesta todo:
- Define los 4-5 servicios
- Red interna entre contenedores
- Volúmenes para datos de PostgreSQL (persistencia)
- Variables de entorno desde `.env`
- Health checks
- Restart policies

### 5.4 — Variables de entorno (.env)

```
# Database
POSTGRES_USER=alpelo
POSTGRES_PASSWORD=<contraseña_segura>
POSTGRES_DB=alpelo_crm

# Backend
DATABASE_URL=postgresql+asyncpg://alpelo:<password>@postgres:5432/alpelo_crm
SECRET_KEY=<clave_jwt_segura>
CORS_ORIGINS=https://tudominio.com

# WhatsApp
WHATSAPP_PHONE_ID=<tu_phone_number_id>
WHATSAPP_TOKEN=<tu_access_token>
WHATSAPP_VERIFY_TOKEN=<tu_webhook_verify_token>

# Claude AI
ANTHROPIC_API_KEY=sk-ant-<tu_clave>

# Frontend
VITE_API_URL=https://tudominio.com/api
```

### 5.5 — Pasos de deploy en VPS

1. **Comprar VPS** — DigitalOcean o Hetzner ($5-10 USD/mes, 2GB RAM mínimo)
2. **Apuntar dominio** — Configurar DNS A record al IP del VPS
3. **Instalar Docker + Docker Compose** en el VPS
4. **Clonar el repo** desde GitHub
5. **Crear archivo `.env`** con las variables reales
6. **`docker compose up -d`** — Levanta todo
7. **Configurar SSL** con certbot/Let's Encrypt
8. **Configurar backup automático** de PostgreSQL (cron + pg_dump)

**Tiempo estimado: 1 semana**
