FEATURE 1: BOOKING ONLINE — Página pública de reservas
============================================================================

PRIORIDAD: CRÍTICA — Sin esto no se puede ir a producción.
REFERENCIA: Weibook tiene esto y es lo #1 que usan los negocios.

── QUÉ ES ──

Una página web pública (sin login) donde los clientes del negocio
pueden agendar sus propias citas 24/7. El dueño pone el link en
Instagram, Facebook, Google Business, WhatsApp, y los clientes
reservan solos.

URL: https://plexifystudio-projects.github.io/AlPelo-CRM/book/alpelo
     O con dominio: book.plexifystudio.com/alpelo

Cada tenant tiene su propio slug → su propia página de booking.

── FLUJO DEL CLIENTE (5 pasos) ──

PASO 1: SERVICIOS
- El cliente ve las categorías de servicios del negocio
- Cards con: nombre, precio (COP), duración, descripción corta
- Agrupados por categoría (Barbería, Manicure, Spa, etc.)
- Click en uno para seleccionar (puede seleccionar varios)
- Diseño: cards con ícono, precio destacado, hover con elevación

PASO 2: PROFESIONAL
- Ve los profesionales que ofrecen ese servicio
- Cards con: FOTO del profesional, nombre, rol, especialidad, rating
- Opción "Sin preferencia" (el sistema asigna automáticamente)
- Si solo hay 1 profesional para ese servicio, se auto-selecciona
- Diseño: cards horizontales con foto circular, nombre bold, badges

PASO 3: FECHA Y HORA
- Calendario visual (próximos 14 días)
- Días sin disponibilidad aparecen grises/deshabilitados
- Al seleccionar día, aparecen los slots disponibles
- Slots calculados desde: horario del staff + citas existentes + duración servicio
- Solo muestra slots REALMENTE libres (no doble-booking)
- Diseño: calendario tipo Google, slots como botones pill con hora

PASO 4: DATOS DEL CLIENTE
- Nombre completo (requerido)
- Teléfono WhatsApp (requerido) — para confirmación automática
- Email (opcional)
- Notas (opcional) — "Quiero corte estilo fade", etc.
- Si el teléfono ya existe en la BD → auto-completa nombre

PASO 5: CONFIRMACIÓN
- Resumen visual de todo:
  - Servicio(s) seleccionado(s) con precio
  - Profesional con foto
  - Fecha y hora
  - Total a pagar
- Botón "Confirmar reserva"
- Al confirmar:
  1. Se crea la cita en Appointment (status="confirmed", created_by="client")
  2. Se crea/actualiza el Client en la BD
  3. Se envía confirmación por WhatsApp (automation "appointment_created")
  4. Se muestra pantalla de éxito con resumen + "Agregar a calendario"

── BACKEND (3 endpoints públicos, SIN autenticación) ──

GET /api/public/book/{slug}
  → Retorna: tenant info (nombre, dirección, horario, logo),
    servicios activos con categoría/precio/duración,
    staff activos con nombre/foto/rol/rating/especialidad,
    horario de atención del negocio

GET /api/public/book/{slug}/availability
  Query params: service_id, staff_id (opcional), date (YYYY-MM-DD)
  → Retorna: lista de slots disponibles ["09:00","09:45","10:30",...]
  → Calcula: horario staff - citas existentes - duración servicio
  → Si staff_id=null, retorna slots de CUALQUIER staff disponible

POST /api/public/book/{slug}/reserve
  Body: { service_id, staff_id, date, time, client_name, client_phone,
          client_email?, notes? }
  → Crea Appointment (tenant_id del slug, status=confirmed, created_by=client)
  → Crea/actualiza Client (si teléfono no existe, crea nuevo)
  → Retorna: { success, appointment_id, confirmation_message }

============================================================================
NOTAS TÉCNICAS
============================================================================

- El booking page NO está dentro de CRMShell (no requiere auth)
- Se registra como ruta pública en App.jsx, igual que las landing pages
- Los endpoints de booking son /api/public/book/... (sin auth middleware)
- Las fotos se guardan como base64 en la BD (sin storage externo)
- El cálculo de disponibilidad debe considerar:
  - Horario del staff (StaffSchedule)
  - Días off (StaffDayOff)
  - Citas existentes (Appointment con status != cancelled)
  - Duración del servicio seleccionado
  - Buffer de 0 min entre citas (el slot debe caber completo)
- La confirmación por WhatsApp usa la automation "appointment_created"
  que ya está configurada y aprobada por Meta
- Multi-location: el booking page muestra las sedes disponibles
  si el tenant tiene 2+, el cliente elige la sede primero

============================================================================
REGLAS DE SCSS (aplicar en TODA la sesión)
============================================================================

1. BEM obligatorio: .block__element--modifier
2. Variables de _variables.scss — NUNCA hardcodear colores
3. Mixins de _mixins.scss para responsive, flex, transiciones
4. NO max-width en contenedores (full width siempre)
5. Animaciones: $ease-out-expo, $ease-spring, staggered
6. Glassmorphism: $glass-bg, backdrop-filter: blur()
7. Shadows: $shadow-xs → $shadow-2xl progresivos
8. Hover states: translateY(-2px) + shadow elevation
9. Focus rings: 0 0 0 3px $color-primary-ghost
10. Loading: skeleton shimmer, no spinners genéricos
11. Transiciones: $transition-base (200ms), $transition-slow (350ms)
12. Border radius: $border-radius-md (12px) para cards
13. Espaciado: $spacing-md (16px), $spacing-lg (24px)
14. Tipografía: $font-size-sm (14px) para body, $font-size-xs (12px) para meta
15. Mobile-first: diseñar para 320px, expandir con media queries
